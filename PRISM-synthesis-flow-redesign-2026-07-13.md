## 1. Executive Summary

PRISM 동아리 팀(3~10명)은 Synthesis로 아이디어를 합성하려 할 때 어떤 가지 조합을 넣을지 스스로 경우의 수를 계산해야 하고, 중간 산출물이 보이지 않아 결과가 나빠도 원인(조합 선택 탓인지 알고리즘 탓인지)을 진단할 수 없다. 그 결과 팀 전체 합성 시도가 10회 미만에 그쳤고 "건졌다" 싶은 결과는 0회로, 도구가 팀에 정착하지 못하고 있다. 근본 원인은 조합 판단에 필요한 정보·기준이 알고리즘 내부에만 존재하는 것(RC-1), 2ⁿ 조합 탐색을 전부 인간에게 전가하는 것(RC-2), 4단계 일괄 실행이라 개입·진단 지점이 없는 것(RC-3)이다. 목표는 "아이디어를 쌓고 → 전체 선택 → 실행 → 판단 지점(차원 거름망, 조합 선택)에만 인간이 개입하고 기계적 반복은 자동"인 플로우로, 오늘(2026-07-13) 17시 비대면 팀 회의에서 실사용하는 것이 1차 관문이다. 요구사항 7건 전원 Must이며, X 품질 프롬프트 잔이슈는 이번에 명시적으로 제외한다(Won't).

## 2. Problem — 문제 정의

### 2.1 관찰된 현상 (사실만)

- 팀 전체 합성 시도 누적 10회 미만, 그중 "건졌다" 싶은 결과 0회 (사용자·팀원 실사용 경험)
- 시도가 적은 이유 자체가 "어떤 가지 조합을 선택해야 할지 몰라서" — 문제가 사용량을 억제하는 악순환
- 팀원 여러 명의 아이디어가 쌓이면 무엇과 무엇을 합성할지 판단하지 못함
- 합성 실행 시 결과만 표시되고 중간 산출물(차원, 직교 쌍)은 UI에서 확인 불가 (코드 확인: 4단계가 서버 내부 일괄 실행)
- 결과 품질이 낮을 때, 조합 선택을 잘못한 탓인지 알고리즘이 나쁜 탓인지 팀이 구분하지 못함

### 2.2 Problem Statement

> **PRISM 팀원들**이 **Synthesis로 아이디어를 합성하려 할 때**, **(입력) 가지 조합을 스스로 계산해야 하고 (실행) 중간 산출물이 보이지 않으며 (결과) 낮은 품질의 원인을 귀속할 수 없는 문제**를 겪고 있으며, 그 결과 **도구에 대한 신뢰와 사용 경험이 쌓이지 않아 팀에 정착하지 못하는 손실**이 발생하고 있다.

### 2.3 문제 분해 (MECE)

**분해 축**: 합성 프로세스 단계별 (입력 → 실행 → 결과)

| 구분 | 문제 | 심각도 | 비고 |
|---|---|---|---|
| 입력 | 합성할 가지 조합을 인간이 경우의 수로 직접 계산해야 함 | 상 | 사용 포기의 직접 원인 |
| 실행 | 중간 산출물(차원·직교 쌍)이 비가시 — 개입 불가, 단계 간 내용 휘발 | 상 | UI에도 미노출 |
| 결과 | 품질 불만족 시 원인(조합 vs 알고리즘) 귀속 불가 | 중 | X 품질 자체는 "판단할 만큼 못 써봄" |

## 3. Root Cause — 근본 원인

### 3.1 Why 체인

**대상 문제**: 어떤 가지 조합을 넣을지 몰라 합성을 시도하지 못함 (입력)

1. 왜? → 좋은 조합을 고를 판단 재료가 없다 (사실 — 사용자 선택: 정보 부족)
2. 왜? → 가지 간 관계(차원·유사·대립)를 보여주는 것이 UI에 없다 (사실 — 코드 확인)
3. 왜? → 그 정보(차원, 직교성)가 알고리즘 내부 개념으로만 존재하고 사용자에게 노출되지 않는다 (사실) → **RC-1**

병행 갈래: "좋은 조합"의 기준 자체를 사용자가 모른다 (사실 — 사용자 선택: 기준 부재) → RC-1에 병합 / 가지 n개면 조합이 2ⁿ개라 전수 검토 불가 (사실 — 사용자 선택: 경우의 수 폭발) → **RC-2**

**대상 문제**: 결과가 나빠도 원인을 진단할 수 없음 (실행·결과)

1. 왜? → 중간 산출물을 볼 수 없다 (사실)
2. 왜? → 4단계가 서버 내부에서 일괄 실행되고 최종 결과만 반환한다 (사실 — 코드 확인) → **RC-3**

### 3.2 핵심 원인 요약

| ID | 근본 원인 | 근거 | 상태 |
|---|---|---|---|
| RC-1 | 조합 판단 정보·기준(차원, 직교성)이 알고리즘 내부에만 존재하고 사용자에게 미노출 | 사용자 선택(정보 부족·기준 부재) + 코드·UI 확인 | 사실 |
| RC-2 | 2ⁿ 조합 탐색을 도구가 돕지 않고 인간에게 전가 | 사용자 선택(경우의 수 폭발) + 현행 UI 구조 | 사실 |
| RC-3 | 4단계 일괄 실행 — 인간 개입 지점·진단 지점 부재 | 코드 확인 (`app/api/synthesize` → `lib/synthesis-engine.ts`) | 사실 |

참고: "책임 부담(남의 아이디어를 버리는 셀렉션이 부담)"은 가설로 제시했으나 사용자가 선택하지 않음 — 기각.

## 4. Intent — 목표

### 4.1 To-Be (목표 상태)

아이디어가 생각나는 대로 쌓인다. 모였다 싶으면 **전체 선택**(일부는 개별 제외)하고 한 번 실행한다. 도구가 차원을 추출해 보여주면 사람이 **거름망**으로 거르고, 도구가 엮어 온 **조합 후보** 중에서 합성할 것들을 고른다. 선택된 조합들의 합성은 **루프로 자동 처리**되고, 모든 단계 산출물과 합성 결과는 휘발되지 않고 화면에 남아 다른 조합을 이어서 합성해볼 수 있다. 운영 원칙: **인간은 판단 지점에만 개입하고, 기계적 반복은 도구가 알아서 돈다.**

### 4.2 성공 기준 (SMART)

| # | 성공 기준 | 측정 방법 | 목표 시점 |
|---|---|---|---|
| G-0 | 2026-07-13 17시 비대면 팀 회의에서 새 플로우로 실사용 | 회의에서 실제 실행 여부 | 오늘 17:00 |
| G-1 | 팀 회의에서 주 1회 이상 실사용 | synthesis_runs 로그 주간 실행 수 | 배포 후 4주 [가설: 기한] |
| G-2 | 사용자의 조합 사전 계산 단계 소멸 — "전체 선택 → 실행"만으로 합성 도달 | 새 플로우 경유 실행 비율 100% | 배포 즉시 |
| G-3 | "건졌다" 싶은 합성 결과 발생 — 월 1건 이상 [가설: 수치] | [가설] 회의에서 팀이 채택해 후속 논의·실행으로 이어진 결과 수 (회의록 기준) | 배포 후 4주 [가설] |
