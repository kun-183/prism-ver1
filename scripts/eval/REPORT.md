## 실패 패턴 4가지 (심판 30명의 지적 종합)

### 1. 직교 쌍 0인데 합성 강행 — 아키텍처 문제 ⭐ 최우선
03에서 2단계(직교성 탐지)가 "직교 쌍 탐지 불가, 생산적 반대 없음"이라 진단했는데도 X가 생성됐다.
원인: 4단계가 **완전 독립 호출**이라 3단계(X 생성)가 2단계의 결과를 모른다. 프롬프트의
"직감이 모두 동일(직교 쌍 0)하면 거부" 규칙을 지킬 정보가 3단계에 없다.

### 2. contribution이 실제 X와 무관한 허구 매핑
08에서 contribution 6개 중 3개가 X 본문에 없는 요소를 가리켰다.
원인: 4단계(기여 추적)도 독립 호출이라 **실제 생성된 X를 보지 못하고** 자기가 상상한 X의
기여를 매핑한다. 구조적으로 fiction이 될 수밖에 없다. (04에서는 거부됐는데 contribution이 채워짐)

### 3. diversity_warning 4중 중복·자기모순
4단계 각각의 경고를 `joinWarnings`가 " / "로 이어붙여, 같은 얘기 4번 + 수치 모순(차원 4 vs 3)이
그대로 사용자에게 노출된다(03·04·06·09). 반대로 07에서는 정작 필요한 과집중 경고가 4단계 모두에서 누락.

### 4. 합성이 아니라 이어붙이기(평균화) 경향
합격한 케이스에서도 반복 지적: X가 "~하되 ~하고 ~하자" 식으로 입력들을 접속사로 병렬 결합(01·05·08·10).
제약2(유사도 분산성 — 어떤 직감과는 강하게, 어떤 직감과는 약하게)가 사실상 작동하지 않는다.
입력이 5개 이상이면 일부를 조용히 버린다(08, 제약1 위반).

## 권장 수정 (우선순위순)

1. **코드 가드 (5분, route.ts)**: 2단계 결과에서 `orthogonal_pairs.length === 0`이면
   `synthesis_possible: false`로 강제. 패턴 1을 코드 레벨에서 즉시 차단.
2. **단계 의존성 재설계 (핵심)**: 1·2단계는 병렬 유지 → 3단계(X 생성)에 1·2단계 결과를 입력으로 전달
   → 4단계(기여 추적)에 **실제 X**를 전달. 지연은 1왕복 → 3왕복으로 늘지만 패턴 1·2가 근본 해결된다.
   PRD의 "독립 호출" 설계는 단계 간 오염 방지가 목적이었으나, 3·4단계는 앞 단계 산출물이 곧 입력이어야 하는 단계다.
3. **경고 정리 (10분)**: 다양성 판정 책임을 1단계 하나에만 부여하고 나머지 단계의 경고는 버리기
   (또는 코드에서 중복 제거). 패턴 3 해결.
4. **프롬프트 보강 (제약2·1)**: X 생성 단계에 자체 점검 추가 —
   "X가 입력들을 접속사로 병렬 연결한 문장인가? 그렇다면 재생성" +
   "N≥5일 때 각 주요 차원이 X에 닿는지 확인, 안 닿는 차원이 있으면 재생성".
   거부 시 contribution을 빈 객체로 강제하는 것도 코드에서 처리.
